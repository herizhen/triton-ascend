# Architecture Design and Core Features

## 1. Logical Architecture

![Image](./figures/architectural_diagram.png)
**Triton-Ascend Architecture Description**

**Core Components:**

- **`Ascend language extension`**: Triton language extension adapted for Ascend
- **`compiler`**: Triton compiler adapted for Ascend
- **`driver`**: Device driver interface adapted for Ascend

**Component Functions:**

- **`Ascend language extension`**
  Introduces syntax and semantic extensions for the Ascend NPU architecture based on the standard Triton language.

- **`compiler`**
  Receives the intermediate representation file `TTIR` (Triton IR) generated by the upper-level Triton compiler and performs a series of transformations adapted for Ascend hardware.

  ```python
  Triton IR â†’ Linalg IR â†’ AscendNPU IR â†’ triton_xxx_kernel.o
  ```

  Triton IR is converted to Linalg IR, and then via the BiSheng Compiler, an executable binary `triton_xxx_kernel.o` targeting the Ascend NPU is generated.

- **`driver`**
  Provides the interface between the Triton runtime and the Ascend software stack (CANN), loading the device-side executable kernel `triton_xxx_kernel.o` generated by the BiSheng Compiler.

## 2. Code Structure

### 2.1 Code Structure Principles

This project extends the standard Triton to support Huawei Ascend NPU (via the CANN software stack). The overall design follows the following **code principles**:

> - **If the modification is target independent**, it should remain in the **Triton core** part (e.g., general modifications to language, runtime);
> - **If the modification is strongly related to Ascend hardware** (target affinitive), it should be placed in **Triton-Ascend**.

### 2.2 Directory Structure and Function Descriptions

**`include/` and `lib/`**

- **Content**: Contains **MLIR Passes**, **Dialects**, and related tools for the Ascend NPU.
- **Purpose**: Used to express and optimize Ascend-specific computation graphs within the MLIR compilation flow.

**`libdevice.py`**

- **Content**: `libdevice` interface adapted for Ascend NPU.
- **Purpose**: Provides low-level implementation support adapted for Ascend NPU hardware, called by Triton operators.

**`backend/compiler.py`**

- **Content**: Main entry point for the `triton-ascend` compiler.
- **Purpose**: Compiles high-level Triton DSL code into **executable binary files** (e.g., `.o` files) that can run on the Ascend NPU.

**`backend/driver.py`**

- **Content**: `triton-ascend` driver module.
- **Purpose**: Loads and launches the compiled executable binary.

##### 3.2.2.1 TritonToStructured

Handles integer division and modulo in pointer expressions and mask expressions. By using dimension elevation, it removes integer division and modulo operations and regenerates OPs like load/store.

| Converter                | Function  | Limitations |
| ------------------------ | -------------------------- | ------------------------- |
| RewriteAddPtrOp          | Analyzes pointer expressions (`AddPtrOp`) in operations like `tl.load`, `tl.store`. Decomposes the original pointer offset calculation and models it into a `PtrState` object containing specific offset information for each dimension (axis). For example, for an expression like `ptr + x // 1024 * 4096 + x % 1024 * 4 + y`, it analyzes the contributions and relationships of the `x` and `y` axes.                   | 1. The original iteration axis involved (e.g., `x`) must be divisible by the split axis (e.g., `1024`).<br>2. The external `XBLOCK` size must be an integer multiple or a divisor of the split axis `divisor`.                               |
| CreateAddpr              | Based on the analyzed `PtrState` object, reconstructs a new `AddPtrOp` pointer calculation operation. The newly generated pointer expression will eliminate the integer division (`//`) and modulo (`%`) operations from the original expression.                                                                       | Depends on a valid `PtrState` successfully generated by `RewriteAddPtrOp`.                                                                                                     |
| RewriteLoadOp            | Analyzes the mask expression in the `tl.load` operation. Decomposes complex mask conditions containing integer division/modulo into a `MaskState` object containing boundary information for each dimension. For example, for `mask = x // 1024 < 8 and x % 1024 < 1024 and y < 4`, it analyzes the independent constraints for each dimension.                                                                     | 1. ##### 3.2.2.2 TritonToUnstructured

| No. | Pass / Converter                              | Description  |
|------|-------------------------------------------|-----------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| 1    | discrete-mask-access-conversion           | Analyzes and transforms memory access patterns based on discrete index masks (Discrete Mask) in Triton (e.g., `triton.language.load` with non-contiguous mask), preparing for the subsequent expansion of discrete axes into loops. This Pass identifies irregular or sparse access patterns that cannot be efficiently handled by the backend hardware. |
| 2    | triton-to-unstructured           | Converts tensor operations containing discrete axes (Discrete Axes), identified by `discrete-mask-access-conversion`, into scalar memory accesses based on explicit scalar loops. |
| 3    | bubble-up-operation                       | Primarily optimizes the upward movement of `extract op/extract_slice` sequences. This can optimize data locality and, in some scenarios, eliminate unnecessary loops generated after the conversion, thereby improving the execution efficiency of the generated code. |

###### 3.2.2.2.1 discrete-mask-access-conversion

| Converter Name                  | Description|
|----------------------------|--------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| DiscreteMaskStoreConversion | First performs mask analysis. If the mask analysis result is non-contiguous, the original store operation is transformed into the following sequence:<br>1. load (load the content of the target storage address)<br>2. select (select between the target storage content and the value to be stored based on the mask)<br>3. store (store the result of the select back to the target address) |
| DiscreteMaskLoadConversion  | First performs mask analysis. If the mask analysis result is non-contiguous, the original load operation is transformed into the following sequence:<br>1. load (load all content of the source tensor)<br>2. select (select content from the source tensor based on the mask; masked parts are set to the `other` value)                     |
| DiscreteMaskAtomicAddConversion | First performs mask analysis. If the mask analysis result is non-contiguous, the original atomic_add operation is transformed into the following sequence:<br>1. select (select the value based on the mask; masked parts are set to 0)<br>2. atomic_add (regenerate the atomic_add operation using the result after select) |
