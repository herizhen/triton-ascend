# Architecture Design and Core Features

## 1. Logical Architecture

![Image](./figures/architectural_diagram.png)
**Triton-Ascend Architecture Description**

**Core Components:**

- **`Ascend language extension`**: Triton language extension adapted for Ascend
- **`compiler`**: Triton compiler adapted for Ascend
- **`driver`**: Device driver interface adapted for Ascend

**Component Functions:**

- **`Ascend language extension`**
  Introduces syntax and semantic extensions targeting the Ascend NPU architecture, built upon the standard Triton language.

- **`compiler`**
  Receives the intermediate representation `TTIR` (Triton IR) generated by the upper-level Triton compiler and performs a series of transformations adapted for Ascend hardware.

  ```python
  Triton IR → Linalg IR → AscendNPU IR → triton_xxx_kernel.o
  ```

  Triton IR is converted to Linalg IR, which is then processed by the BiSheng Compiler to generate an executable binary `triton_xxx_kernel.o` for the Ascend NPU.

- **`driver`**
  Provides the interface between the Triton runtime and the Ascend software stack (CANN), loading the device-side executable kernel `triton_xxx_kernel.o` generated by the BiSheng Compiler.

## 2. Code Structure

### 2.1 Code Structure Principles

This project extends the standard Triton to support Huawei Ascend NPU (via the CANN software stack). The overall design follows these **code principles**:

> - **If the modification is target independent**, it should remain in the **Triton core** part (e.g., general modifications to language, runtime);
> - **If the modification is strongly related to Ascend hardware** (target affinitive), it should be placed in **Triton-Ascend**.

### 2.2 Directory Structure and Function Descriptions

**`include/` and `lib/`**

- **Content**: Contains **MLIR Passes**, **Dialects**, and related tools for the Ascend NPU.
- **Purpose**: Used to express and optimize Ascend-specific computation graphs within the MLIR compilation flow.

**`libdevice.py`**

- **Content**: `libdevice` interface adapted for the Ascend NPU.
- **Purpose**: Provides underlying implementation support adapted for Ascend NPU hardware, callable by Triton operators.

**`backend/compiler.py`**

- **Content**: Main entry point for the `triton-ascend` compiler.
- **Purpose**: Compiles high-level Triton DSL code into **executable binary files** (e.g., `.o` files) that can run on the Ascend NPU.

**`backend/driver.py`**

- **Content**: `triton-ascend` driver module.
- **Purpose**: Loads and launches the compiled executable binary.

##### 3.2.2.1 TritonToStructured

Handles integer division and modulo operations in pointer expressions and mask expressions. By increasing dimensions, it removes integer division and modulo operations and regenerates OPs like load/store.

| Converter | Function | Limitations |
| ------------------------ | -------------------------- | ------------------------- |
| RewriteAddPtrOp | Analyzes pointer expressions (`AddPtrOp`) in operations like `tl.load`, `tl.store`. Decomposes the original pointer offset calculation and models it as a `PtrState` object containing specific offset information for each dimension (axis). For example, for an expression like `ptr + x // 1024 * 4096 + x % 1024 * 4 + y`, it analyzes the contribution and relationship of the `x` and `y` axes. | 1. The original iteration axis involved (e.g., `x`) must be divisible by the split axis (e.g., `1024`).<br>2. The external `XBLOCK` size must be an integer multiple or a divisor of the split axis `divisor`. |
| CreateAddpr | Based on the analyzed `PtrState` object, reconstructs a new `AddPtrOp` pointer calculation operation. The newly generated pointer expression eliminates the integer division (`//`) and modulo (`%`) operations from the original expression. | Depends on a valid `PtrState` successfully generated by `RewriteAddPtrOp`. |
| RewriteLoadOp | Analyzes the mask expression in the `tl.load` operation. Decomposes complex mask conditions containing integer division/modulo and models them as a `MaskState` object containing boundary information for each dimension. For example, for `mask = x // 1024 < 8 and x % 1024 < 1024 and y < 4`, it analyzes the independent constraint conditions for each dimension. | 1. The original iteration axis involved (e.g., `x`) must be divisible by the split axis (e.g., `1024`).<br>2. The external `XBLOCK` size must be an integer multiple or a divisor of the split axis `divisor`. |
| BuildMask | Based on the analyzed `MaskState` object, reconstructs a new mask expression. The new mask eliminates the integer division (`//`) and modulo (`%`) operations from the original expression. | Only handles `MaskState` generated by `RewriteLoadOp` or `RewriteStoreOp`. Cannot handle arbitrarily complex, non-canonical mask expressions. |
| CreateLoad | Uses the new pointer expression generated by `CreateAddpr` and the new mask expression generated by `BuildMask` to recreate (replace) the original `tl.load` operation, completing the instruction rewrite. | Depends on the successful execution of prerequisite steps like `RewriteAddPtrOp`, `CreateAddpr`, `RewriteLoadOp`, and `BuildMask`. |
| RewriteStoreOp | Analyzes the mask expression in the `tl.store` operation. Its function is similar to `RewriteLoadOp`, decomposing complex mask conditions containing integer division/modulo and modeling them as a `MaskState` object. | Same as `RewriteLoadOp`. |
| CreateStore | Uses the new pointer expression generated by `CreateAddpr` and the new mask expression generated by `BuildMask` to recreate (replace) the original `tl.store` operation, completing the instruction rewrite. | Depends on the successful execution of prerequisite steps like `RewriteAddPtrOp`, `CreateAddpr`, `RewriteStoreOp`, and `BuildMask`. |
| RewriteAtomicRWMOp | Handles pointer issues in atomic read-write-modify operations (e.g., `atomic.add`, `atomic.max`). | Typically inherits the same limitations as `RewriteAddPtrOp`. May not support certain special, non-contiguous, or conditional atomic operation patterns. |
| RewriteAtomicCASOp | Handles pointer linearization in atomic compare-and-swap operations (`atomic.cas`). Analyzes its pointer expression and eliminates integer division and modulo operations by increasing dimensions to meet the addressing requirements of hardware atomic instructions. | |
| RewriteWhile | Handles pointer accumulation operations within `while` loop bodies. | Does not support complex pointer path transformations involving conditional branches (`if`) within the loop body. |
| RewriteFor | Handles pointer accumulation operations within `for` loop bodies. | |

##### 3.2.2.2 TritonToUnstructured

| No. | Pass / Converter | Description |
|------|-------------------------------------------|-----------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| 1    | discrete-mask-access-conversion | Analyzes and converts memory access patterns based on discrete index masks in Triton (e.g., `triton.language.load` with a non-contiguous mask), preparing for the subsequent expansion of discrete axes into loops. This pass identifies irregular or sparse access patterns that cannot be efficiently handled by the backend hardware. |
| 2    | triton-to-unstructured | Converts tensor operations containing discrete axes identified by `discrete-mask-access-conversion` into scalar memory accesses based on explicit scalar loops. |
| 3    | bubble-up-operation | Primarily optimizes the upward movement of `extract op/extract_slice` sequences. This can improve data locality and, in some scenarios, eliminate unnecessary loops generated after the conversion, thereby improving the execution efficiency of the generated code. |

###### 3.2.2.2.1 discrete-mask-access-conversion

| Converter Name | Description|
|----------------------------|--------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| DiscreteMaskStoreConversion | First performs mask analysis. If the mask analysis result is non-contiguous, the original store operation is transformed into the following sequence:<br>1. load (load the content of the target storage address)<br>2. select (select between the target storage content and the value to be stored based on the mask)<br>3. store (store the result of the select back to the target address) |
| DiscreteMaskLoadConversion | First performs mask analysis. If the mask analysis result is non-contiguous, the original load operation is transformed into the following sequence:<br>1. load (load all content of the source tensor)<br>2. select (select content from the source tensor based on the mask; masked-out parts are set to the `other` value) |
| DiscreteMaskAtomicAddConversion | First performs mask analysis. If the mask analysis result is non-contiguous, the original atomic_add operation is transformed into the following sequence:<br>1. select (select the value based on the mask; masked-out parts are set to 0)<br>2. atomic_add (regenerate the atomic_add operation using the result after the select) |
